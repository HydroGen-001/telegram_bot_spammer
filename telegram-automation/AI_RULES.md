# AI_RULES — Правила разработки и архитектуры

## Общие принципы

1. **Стабильность** — приоритет над скоростью
2. **Надёжность** — обработка всех ошибок
3. **Простота** — минимальная сложность кода
4. **Безопасность** — минимизация риска блокировок

## Архитектурные правила

### Модульность
- Каждый модуль отвечает за одну функцию
- Модули не зависят друг от друга напрямую
- Все зависимости через ядро (core)

### Асинхронность
- Все I/O операции асинхронные
- Использовать `asyncio` для всех сетевых запросов
- Не блокировать основной цикл

### Конфигурация
- Все настройки в JSON файлах
- Валидация конфигурации при запуске
- Значения по умолчанию для всех параметров

### Логирование
- Логировать все действия
- Уровни: DEBUG, INFO, WARNING, ERROR
- Ротация логов (10MB, 5 файлов)

## Правила анти-бан защиты

### Лимиты (актуальные для Telegram 2024-2026)

| Параметр | Значение |
|----------|----------|
| Сообщений в сутки | 50-100 (новый аккаунт), 200-300 (прогретый) |
| Сообщений в час | 20-30 |
| Задержка между сообщениями | 30-120 секунд |
| Пауза после N сообщений | 5-10 минут после 10 сообщений |

### Рандомизация
- Все задержки случайные в диапазоне
- Шаблоны сообщений выбираются случайно
- Порядок чатов перемешивается

### Поведение
- Имитация человеческого поведения
- Избегание паттернов бота
- Учёт часовых поясов

## Правила работы с базой данных

### SQLite оптимизация
- WAL режим для лучшей производительности
- Индексы на часто используемых полях
- Пакетная вставка записей

### Схема данных
- `messages` — отправленные сообщения
- `incoming` — входящие сообщения
- `stats` — статистика по дням

## Правила обработки ошибок

1. **Сетевые ошибки** — повторная попытка с экспоненциальной задержкой
2. **Ошибки API** — логирование и продолжение работы
3. **Критические ошибки** — остановка с уведомлением
4. **FloodWait** — соблюдение времени ожидания от Telegram

## Безопасность данных

- Не хранить токены в коде
- Использовать `.env` для чувствительных данных
- Сессии хранить в защищённой папке
- Не логировать чувствительные данные

## Тестирование

- Модульные тесты для каждого модуля
- Интеграционные тесты с тестовым аккаунтом
- Проверка лимитов перед запуском

## Версионирование

- SemVer для версий
- Changelog для изменений
- Миграции БД при изменении схемы
